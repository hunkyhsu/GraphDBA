# ADR 002: adopt subquery limit for MCP servers

**Date:** 2026-05-03
**Status:** Accepted

## Context
The Read Probe MCP Server allows AI agents (specifically the Diagnostic and Validation agents) to execute read-only `SELECT` queries against the target PostgreSQL database to gather live diagnostic data. Because LLMs can hallucinate or write poorly optimized queries (e.g., `SELECT * FROM massive_log_table;`), we must enforce a hard limit on the number of rows returned. This prevents:
1. Python Out-Of-Memory (OOM) crashes on the MCP server.
2. Token context window overflow for the LLM.
3. Unnecessary I/O and memory strain on the target database.

The architectural challenge is: **How do we reliably inject a row limit into unpredictable, AI-generated SQL that might already contain its own `LIMIT`, `ORDER BY`, or nested subqueries?**

## 2. Considered Options

### Option A: "Detect and Inject" (Lexical/AST Parsing)
This approach involves analyzing the AI's query string in Python to see if a `LIMIT` clause already exists.
* If no limit exists, append `LIMIT 100`.
* If a limit exists (e.g., `LIMIT 5000`), parse the number, evaluate if it is greater than our maximum, and overwrite it if necessary.

### Option B: "Subquery Wrapping" (The DBA Standard)
This approach ignores the internal contents of the AI's query and unconditionally wraps it in an outer subquery:
* `SELECT * FROM ( <AI_GENERATED_QUERY> ) AS mcp_limited_query LIMIT 100;`

## 3. Decision Outcome
**Decision:** We will use **Option B (Subquery Wrapping)**.

### Rationale
1. **Elimination of Parsing Complexity:** Option A requires either highly brittle Regular Expressions (which fail on edge cases like `WHERE text_column = 'limit 1000'`) or full Abstract Syntax Tree (AST) parsing in Python (using heavy libraries like `pglast`). Option B requires only basic string manipulation (stripping trailing semicolons).
2. **Delegating to the C-Kernel:** Option B pushes the AST parsing and optimization burden down to the PostgreSQL Query Planner, which is written in highly optimized C. 
3. **Native Optimization:** If the AI generates a query with a large limit (`SELECT * FROM users LIMIT 10000;`), the PostgreSQL query planner evaluates the wrapped query and natively optimizes the execution tree, stopping execution gracefully after the outer 100 rows are reached. 
4. **Absolute Safety:** The subquery method is immune to syntax hallucinations and guarantees the hard limit is enforced at the database protocol layer before data is ever sent over the network to the MCP server.

## Consequences

### Positive
* **Zero Brittleness:** Complete immunity to complex SQL syntax variations generated by the LLM.
* **Lightweight Middleware:** Keeps the Python middleware incredibly fast and CPU-efficient.
* **Network Efficiency:** Prevents large payloads from leaving the database layer.

### Negative
* **Minor String Pre-Processing:** The middleware must carefully strip trailing whitespace and semicolons from the AI's query before wrapping it, otherwise PostgreSQL will throw a syntax error. (Mitigated by a simple `.strip().rstrip(';')` operation).
